# 材料清单规则 V2.0

生成业务逻辑清单前，按以下顺序读取原材料，确保信息完整。支持 **Extract模式** 和 **Draft模式** 两种模式。

---

## 模式说明

| 模式 | 输入来源 | 适用场景 |
|------|----------|----------|
| **Extract** | HTML源码逆向提取 | 有原型页面 |
| **Draft** | 需求描述正向生成 | 无原型页面，首次设计 |

---

## 一、Extract模式材料清单

### 第一层：业务规格（必读）

| # | 材料 | 路径示例 | 作用 |
|---|------|----------|------|
| 1 | 导航索引 | `page-index.md` 或 `page-index.html` | 页面层级、版本标记、页面列表 |
| 2 | 现有业务逻辑清单 | `doc/V{版本号}/业务逻辑清单_*.md` | 已有版本内容，识别增量变更 |
| 3 | 现有PRD | `doc/V{版本号}/PRD_*.md`（如有） | 需求细节、业务规则补充 |

### 第二层：源码实现（逐页读取）

仅读取本次版本范围内的页面源码：

| 类型 | 路径模式 | 作用 |
|------|----------|------|
| 页面HTML | `pages/{page_name}.html` | 交互逻辑、字段定义、数据结构、跳转目标 |
| 公共JS | `js/common.js` | 全局组件、工具函数 |
| 认证JS | `js/auth.js` | 账号体系、登录态管理 |
| 认证守卫 | `js/auth-guard.js` | 受保护页面列表、拦截逻辑 |
| 公共CSS | `css/common.css` | 全局样式、组件尺寸、响应式断点 |

### 第三层：补充材料（按需读取）

| 材料 | 何时读取 | 作用 |
|------|----------|------|
| 会议纪要/讨论记录 | 用户提到之前的讨论 | 需求变更背景、决策理由 |
| 竞品截图/参考 | 用户提到参考某竞品 | 功能对标、交互参考 |

---

### 页面源码阅读要点

读取每个页面 HTML 时，按以下维度逐项提取：

#### 1. 页面结构

- `<title>` 标签文本
- `.nav-title` 导航栏标题
- 页面主要区域划分（通过 CSS class 和 HTML 结构识别）

#### 2. 可交互元素

重点关注以下 HTML 标签和属性：

| 元素 | 提取内容 |
|------|----------|
| `<button>` / `.btn-*` | 按钮文案、绑定事件 |
| `<input>` / `<textarea>` | type、placeholder、maxlength、id |
| `<select>` / 自定义下拉 | 选项列表、默认选中项 |
| `.tab-*` / `.filter-*` | Tab 切换项、筛选标签 |
| 弹窗/面板触发器 | 触发条件和弹出内容 |

#### 3. JavaScript 逻辑

| 代码模式 | 提取内容 |
|----------|----------|
| `addEventListener('click', ...)` | 点击操作及对应行为 |
| `location.href = '...'` | 页面跳转目标 |
| `history.back()` | 返回导航 |
| `alert('...')` / `toast('...')` | 提示文案 |
| `if (...) { ... } else { ... }` | 条件分支 → 多场景用例 |
| `classList.add/remove/toggle` | 状态切换 → 视觉反馈 |
| 数据对象定义（`var items = [...]`） | 模拟数据结构 |
| `localStorage.setItem/getItem` | 状态持久化 |

#### 4. CSS 视觉反馈

| CSS 模式 | 提取内容 |
|----------|----------|
| `.active` / `.selected` 样式 | 选中态描述（颜色、背景） |
| `@keyframes` 动画 | 动画效果描述 |
| `transition` 属性 | 过渡效果 |
| `display: none` → `display: flex` | 展开/折叠行为 |
| 颜色值（`#ff6034`、`#ee0a24`） | 具体颜色描述 |

#### 5. 页面跳转关系

- 所有 `href="..."` 链接
- 所有 `location.href = '...'`
- 所有 `onclick` 中隐含的跳转

---

### 公共模块阅读要点

#### auth.js / auth-guard.js（若有）

- 用户数据结构（字段、默认值）
- 登录/认证方法签名
- `getCurrentUser()` 返回结构
- 测试账号列表（若有）
- 受保护页面列表
- 拦截行为（跳转到哪个登录页）

#### common.css

- `.phone-frame` 尺寸
- `.status-bar` 高度
- `.sub-nav` padding-top
- 响应式断点
- 全局颜色值

---

### 信息整合规则

1. **源码为准** — 功能用例表严格基于源码实现，不臆测未实现的功能
2. **源码补充细节** — 代码中有但未被主动提及的交互细节，全部纳入用例
3. **跳转关系完整** — 每个跳转都记录到跳转关系表中
4. **版本标记对齐** — 文档中的页面范围必须与导航索引的版本标记一致

## 二、Draft模式材料清单

### 第一层：需求输入（用户交互）

按 `rules/requirement-collection.md` 的流程收集：

| # | 收集项 | 收集方式 | 作用 |
|---|--------|----------|------|
| 1 | HMW问题陈述 | 用户口述 → 重述 | 定义问题边界 |
| 2 | 澄清问题答案 | `AskUserQuestion` 工具 | 明确用户/标准/约束 |
| 3 | 功能方向变体 | 生成5-8个变体 → 用户反馈 | 探索设计空间 |
| 4 | 选定方向 | 用户选择或组合 | 确定页面范围 |

### 第二层：上下文扫描（Codebase-Aware）

参考 idea-refine 的 codebase 扫描要求：

| 扫描维度 | 搜索模式 | 识别内容 |
|----------|----------|----------|
| 现有相关页面 | `{相关模块}*.html` | 可参考的页面结构 |
| 状态流转定义 | `status.*` | 状态枚举、状态颜色 |
| 表单提交模式 | `submit.*` / `form.*` | 可复用的表单结构 |
| 弹窗组件 | `popup.*` / `modal.*` | 可复用的弹窗样式 |
| 后台审批模式 | `review.*` / `approve.*` | 可复用的审批流程 |
| 图片上传 | `upload.*` / `image.*` | 可复用的上传组件 |

### 第三层：现有文档参考（可选）

| 材料 | 路径示例 | 作用 |
|------|----------|------|
| 现有PRD | `doc/V{上一版本}/PRD_*.md` | 了解现有功能设计 |
| 现有业务逻辑清单 | `doc/V{上一版本}/业务逻辑清单_*.md` | 了解现有字段定义 |
| 设计规范 | `prototype-design` skill | 确保UI风格一致 |

### Draft模式阅读要点

#### 上下文扫描

使用 `Glob`、`Grep`、`Read` 扫描现有代码库：

**扫描结果记录格式：**

```markdown
### 可复用组件

| 组件 | 来源文件 | 复用方式 |
|------|----------|----------|
| 状态标签 | `order_detail.html` | 售后状态复用订单状态标签样式 |
| 确认弹窗 | `payment.html` | 提交弹窗复用支付确认弹窗结构 |
| 图片上传 | `review.html` | 凭证上传复用评价页图片上传组件 |

### 现有约束

| 约束 | 来源 | 影响 |
|------|------|------|
| 订单状态不可逆 | `order_status` 定义 | 售后状态需独立状态机 |
| 仅支持3张图片 | `review.html` 限制 | 凭证上传同样限制3张 |
```

## 三、模式对比

| 维度 | Extract模式 | Draft模式 |
|------|-------------|-----------|
| **输入来源** | HTML源码 | 需求描述 + codebase扫描 |
| **用例生成** | 从代码逆向提取 | 从设计意图正向推导 |
| **预期结果** | 具体到源码实现 | 设计意图，标注 `[草案]` |
| **数据来源** | 源码中的数据结构 | 不确定时标注 `[待确认]` |
| **假设标注** | 无（源码已实现） | 每页标注 `[必验]`/`[风险]` |
| **Not Doing** | 无（源码已定义） | 必须明确排除功能 |

## 四、常见错误

| 错误 | 正确做法 |
|------|----------|
| 只看HTML不看JS | JS包含核心交互逻辑，必须完整阅读 |
| 跳过CSS部分 | CSS提供视觉反馈描述（颜色、动画） |
| 读取所有页面源码 | 只读版本范围内的页面 |
| 忽略公共模块 | auth.js的数据结构是关键输入 |
| 凭空编造用例 | 每条用例必须能在源码找到对应实现 |
| Draft模式跳过codebase扫描 | 必须扫描现有组件和约束 |
| Draft模式无假设标注 | 每页必须标注假设清单 |
