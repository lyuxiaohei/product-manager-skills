# 页面章节模板 V3.0

单个页面在业务逻辑清单中的标准结构。支持 **正式版** 和 **草案版** 两种模式。

V3.0 变更：功能用例表从 4 列扩展为 6 列（十字法），关键字段数据来源从 2 列改为 4 列（数据溯源三问）。

---

## 一、正式版页面章节

适用于有原型 HTML 的 extract 模式。

```markdown
### {X}.{Y} {页面名称}（{filename}.html）

![{页面名称}](./screenshots/{filename}.png)  ← 仅用户要求时，默认省略此行


#### {X}.{Y}.1 功能用例表

| # | 场景 | 前置条件 | 操作步骤 | 预期结果 | 异常分支 |
|---|------|----------|----------|----------|----------|
| 1 | 图片轮播 | 页面加载完成 | 左右滑动主图 | 切换图片，指示器更新如"1/3"→"2/3" | — |
| 2 | 选择规格 | 进入商品详情 | 点击规格属性值 | 选中项橙色渐变背景白字（#ff6034→#ee0a24），主图同步切换 | SKU已下架→置灰不可选，显示"已售罄" |
| 3 | 提交订单 | 已选商品、已填地址 | 点击"立即支付" | 跳转支付页（`payment.html`） | 库存不足→Toast"库存不足"并刷新商品信息 |
| 4 | 空状态 | — | 购物车无商品时展示 | 图标+"购物车空空如也"文案，"去逛逛"按钮 | — |
| 5 | 校验失败 | 未勾选协议 | 点击提交 | Toast提示"请阅读并同意用户协议" | — |

#### {X}.{Y}.2 后端业务逻辑增强

| # | 问题 | 确认结果 |
|---|------|----------|
| 1 | 库存不足前端提示策略？ | 库存漏斗策略：<br>- 加购时（宽松）：允许超库存加购<br>- 下单时（严格）：实时校验，不足则禁用提交<br>- 支付时（兜底）：二次校验，不足则提示并退款 |
| 2 | 多设备购物车合并策略？ | （建议：同SKU数量相加，异款追加，冲突取最后修改时间） |

#### {X}.{Y}.3 关键数据来源说明

| 字段 | 从哪来 | 往哪去 | 变了怎么办 |
|------|--------|--------|------------|
| 商品信息（图片/名称/规格/价格） | 品牌商城后台商品管理 — SKU主图+SKU轮播图 | 商品详情展示 + 提交订单API — 用于订单创建 | 后台改价→前端实时刷新（接口重新拉取） |
| 商品上下架状态 | 品牌商城后台商品管理 — 实时查询 | 校验逻辑 — 下架商品不可加购 | 下架→列表移除或标记不可购 |
| 购物车数量 | 用户操作 — 点击加购/修改数量 | 提交订单API — 写入订单明细 | 用户修改→实时更新总价 |
| 总价/总数量 | 前端实时计算 — 基于勾选商品×单价 | 仅展示，提交时后端重新计算校验 | 商品价格变化→自动重算 |
| 优惠券信息 | 品牌商城后台优惠券管理 | 提交订单API — 用于金额扣减 | 优惠券失效→标记不可用并提示 |
```

### 正式版编写规则

| 列 | 规则 |
|-----|------|
| **#** | 页内从 1 递增，不跳号 |
| **场景** | 2-6字名词短语 |
| **前置条件** | 触发条件（状态/数据条件），无特殊条件填"—" |
| **操作步骤** | 动词开头，包含元素名称或按钮文案（用引号） |
| **预期结果** | 具体到文案、颜色值（附十六进制）、动画名、跳转页面文件名 |
| **异常分支** | 格式：`异常条件 → 处理方式`，无异常填"—" |
| **字段** | 字段名 + 括号列举子字段（如 `商品信息（图片/名称）`） |
| **从哪来** | 格式：`来源系统 — 具体说明` |
| **往哪去** | 格式：`目标接口/表 — 用途说明` 或 `仅展示` |
| **变了怎么办** | 格式：`变更触发条件 → 同步策略` |

### 关键数据来源说明的概念

**注意：** "关键数据来源说明"不是指来源于某个代码文件，而是指**页面字段在业务层面的数据来源**，即该字段的值从哪个业务系统或页面环节产生。

**数据溯源三问：**

| 问题 | 含义 | 来源分类 |
|------|------|----------|
| **从哪来** | 字段的值从何产生 | 用户输入 / 接口返回 / 本地下拉 / 前端计算 / 上一页传入 / 系统生成 |
| **往哪去** | 提交到哪里 | 提交到某个API / 写入某张表 / 触发某个流程 / 仅展示不提交 |
| **变了怎么办** | 来源数据变更时的处理 | 实时同步 / 手动刷新 / 不影响已提交数据 / 触发通知 |

### 确定的数据来源

直接填写，不加建议前缀：

```
品牌商城后台商品管理 — SKU规格属性
前端实时计算 — 基于勾选商品数量×单价
用户操作 — 本地购物车数据
上一页面传入 — 待评价订单商品信息
```

### 不确定的数据来源

留白并附建议：

```
品牌商城后台{具体模块}（建议：订单管理 — 订单状态定义）
```

## 二、草案版页面章节

适用于无原型 HTML 的 draft 模式。

```markdown
### {X}.{Y} {页面名称} `[待原型]` （{filename}.html）

> **页面定位**：让用户在小程序端发起售后申请，选择申请类型并上传凭证。
>
> **压力测试结论**：
> - **User Value**：Painkiller — 用户投诉"找不到退款入口"，痛点强烈
> - **Feasibility**：Medium — 可复用订单详情页结构，新增表单组件
> - **Differentiation**：显著提升 — 现状需客服电话申请耗时10分钟，新流程自助2分钟
>
> **假设清单**：
> - 用户愿意在小程序端上传凭证图片 `[必验]` — 验证方式：5用户访谈
> - 用户能区分"仅退款"和"退货退款" `[必验]` — 验证方式：原型可用性测试
> - 审核时效超48h用户会流失 `[风险]` — 验证方式：历史投诉数据抽样
> - 暂不考虑跨仓库退货物流调度 `[暂略]` — V0.2决策

#### {X}.{Y}.1 功能用例表

| # | 场景 | 前置条件 | 操作步骤 | 预期结果 | 异常分支 |
|---|------|----------|----------|----------|----------|
| 1 | 选择类型 | 进入申请页 | 点击"仅退款"或"退货退款" Tab | 选中项橙色背景白字（#ff6034→#ee0a24） `[草案]` | — |
| 2 | 填写原因 | 已选择类型 | 点击下拉框选择申请原因 | 展示选项：质量问题/不满意/其他，选中后关闭下拉 `[草案]` | 无可选项→显示"暂无可用原因" `[草案]` |
| 3 | 上传凭证 | 已选择类型 | 点击"添加图片"按钮 | 打开图片选择器，最多3张，显示缩略图预览 `[草案]` | 超过3张→Toast"最多上传3张" `[草案]` |
| 4 | 提交申请 | 已填必填项 | 点击"提交申请"按钮 | 弹出确认弹窗，点击"确认"跳转进度页（`aftersales_progress.html`） `[草案]` | 未填必填项→Toast"请选择申请原因" `[草案]` |
| 5 | 校验失败 | 未选择申请原因 | 点击提交 | Toast提示"请选择申请原因" | — |
| 6 | 空状态 | 订单无可售后商品 | 进入页面时展示 | 图标+"该订单不支持售后"文案，"返回订单"按钮 `[草案]` | — |

#### {X}.{Y}.2 后端业务逻辑增强

| # | 问题 | 确认结果 |
|---|------|----------|
| 1 | 用户愿意上传凭证图片吗？ | `[必验]` 验证方式：5用户访谈，了解上传意愿和顾虑 |
| 2 | 申请时效限制？ | `[待确认]`（建议：订单完成后7天内可申请） |
| 3 | 审核时效要求？ | `[待确认]`（建议：48小时内，超时自动催办） |
| 4 | 图片数量限制？ | `[待确认]`（建议：最多3张，单张≤5MB） |

#### {X}.{Y}.3 关键数据来源说明

| 字段 | 从哪来 | 往哪去 | 变了怎么办 |
|------|--------|--------|------------|
| 申请类型 | 用户操作 — Tab选择 | 提交售后API — 写入售后单类型字段 | 不可变，提交后不可修改 `[草案]` |
| 申请原因 | 用户操作 — 下拉选择 | 提交售后API — 写入售后单原因字段 | 不可变 `[草案]` |
| 凭证图片 | 用户操作 — 图片上传 | 提交售后API — 上传至图片存储服务 | 不可变 `[草案]` |
| 订单信息（商品/金额/订单号） | 品牌商城后台订单管理（建议：通过订单ID关联查询） `[待确认]` | 展示+校验 — 验证订单可售后状态 | 订单状态变更→重新校验可售后性 `[草案]` |
| 申请状态 | 品牌商城后台售后管理 `[待确认]` | 展示 — 售后进度页展示 | 后台审核变更→前端状态同步 `[草案]` |
```

### 草案版编写规则

| 列 | 规则 |
|-----|------|
| **页面定位** | 一句话：解决什么问题 |
| **压力测试** | 必须包含 User Value/Feasibility/Differentiation 三维度 |
| **假设清单** | 每页至少2条假设，区分 `[必验]`/`[风险]`/`[暂略]` |
| **前置条件** | 标注触发条件，含假设的标注 `[草案]` |
| **预期结果** | 具体且标注 `[草案]`，颜色值、文件名提前定义 |
| **异常分支** | 标注 `[草案]`，推演的异常场景需注明 |
| **从哪来** | 用户操作直接写，外部来源标注 `[待确认]` |
| **往哪去** | 标注目标API或表，不确定的标注 `[待确认]` |
| **变了怎么办** | 标注同步策略，标注 `[草案]` |

## 三、状态标记详解

### 草案标记说明

| 标记 | 使用位置 | 含义 | 转换规则 |
|------|----------|------|----------|
| `[草案]` | 预期结果、异常分支列 | 设计意图，待验证 | 正式版移除，替换为源码实现描述 |
| `[待原型]` | H3 页面标题 | 页面未创建 | 正式版移除 |
| `[待确认]` | 数据来源/业务问题 | 需用户决策 | 正式版替换为确认结果或保留建议格式 |
| `[必验]` | 假设清单/业务问题 | 关键假设，验证失败则流程不可行 | 正式版标注验证结果 |
| `[风险]` | 假设清单/业务问题 | 可能失败的假设，需关注但不阻塞 | 正式版标注风险应对 |
| `[暂略]` | 假设清单/Not Doing | 当前版本有意排除 | 正式版移至Not Doing或移除 |

### 标记使用场景

**功能用例表：**

```markdown
| 1 | 选择类型 | 进入申请页 | 点击Tab | 选中项橙色背景 `[草案]` | — |
| 2 | 提交申请 | 已填必填项 | 点击按钮 | 跳转进度页（progress.html） `[草案]` | 未填必填→Toast提示 `[草案]` |
```

**关键字段数据来源（数据溯源三问）：**

```markdown
| 申请类型 | 用户操作 — Tab选择 `[草案]` | 提交售后API — 写入类型字段 | 不可变 `[草案]` |
| 订单信息 | 品牌商城后台订单管理 `[待确认]` | 展示+校验 `[待确认]` | 状态变更→重新校验 `[草案]` |
```

**业务逻辑增强：**

```markdown
| 1 | 用户愿意上传凭证吗？ | `[必验]` 验证方式：用户访谈 |
| 2 | 申请时效限制？ | `[待确认]`（建议：7天内） |
```

## 四、典型用例类型对照

| 用例类型 | 正式版识别方式 | 草案版识别方式 |
|----------|----------------|----------------|
| 点击操作 | `addEventListener('click')` | 用户动作描述 |
| 输入校验 | `if (!value)` / maxlength | 校验规则设计 |
| Tab切换 | `.tab-*` CSS类 | Tab组件设计 |
| 状态切换 | `classList.toggle` | 开关/复选框设计 |
| 弹窗操作 | overlay + popup 结构 | 弹窗交互设计 |
| 页面跳转 | `location.href` | 跳转目标页面名 |
| 空状态 | `if (data.length === 0)` | 空数据场景设计 |
| 视觉反馈 | `.active` / `@keyframes` | 颜色值、动画描述 |
| 异常分支 | `catch(err)` / `if (!success)` | 十字法纵向推演 |

## 五、截图引用规则

> **默认不添加截图。** 仅当用户主动要求（"加截图"/"截图"/"生成截图"）或传入 `--screenshot` 时，才为页面生成并引用截图。否则页面章节直接从 H3 标题进入功能用例表，不插入任何截图占位。

### 正式版（仅用户要求时）

位于 H3 标题和第一个 H4 之间：

```markdown
### 1.1 申请页（aftersales_apply.html）

![申请页](./screenshots/aftersales_apply.png)


#### 1.1.1 功能用例表
```

- 使用相对路径 `./screenshots/{filename}.png`
- 文件名与页面 HTML 文件名一致
- 截图通过 Chrome CDP 自动生成（375×812，2x）

```markdown

```

**无截图**，页面标题标注 `[待原型]`：

## 六、业务逻辑增强表何时生成

仅在以下情况生成此表：

| 判断条件 | 正式版 | 草案版 |
|----------|--------|--------|
| 跨页面状态同步 | 有源码同步逻辑 | 假设清单标注 |
| 时间窗口限制 | 有校验代码 | `[必验]` 验证规则 |
| 并发/防重策略 | 有防重代码 | `[风险]` 识别风险点 |
| 数据一致性 | 有回滚逻辑 | `[必验]` 标注一致性要求 |
| 业务规则选择 | 有配置代码 | `[待确认]` 附建议 |

## 七、Anti-patterns

| 反模式 | 正确做法 |
|--------|----------|
| 草案预期结果写"显示正确" | 具体到交互元素、颜色、跳转页面 |
| 草案不标注假设 | 每页至少2条假设，区分类型 |
| 草案无 Not Doing | Not Doing ≥ Doing 的 50% |
| 草案页面定位写完整功能描述 | 一句话解决问题 |
| 假设不标注验证方式 | 每条假设必须有验证方式 |
| 正式版保留 `[草案]` 标记 | 转换时移除所有草案标记 |
| 异常分支列写"报错"或"失败" | 具体到异常类型+处理方式（如`库存不足→Toast提示`） |
| 正常用例不写异常分支 | 即使无异常也显式填写"—" |
| 数据来源只写"从哪来" | 三问缺一不可，尤其是"变了怎么办" |
| 前置条件列写操作描述 | 仅写触发条件，不写操作 |
