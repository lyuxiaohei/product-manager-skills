# 草案生成规则 V2.0

定义需求收集后的页面收敛和草案文档生成流程。融合十字法和 6 步落地套路。

---

## 生成流程总览

```
需求收集（Phase 1）
  → 收敛三阶段（聚类 → 压力测试 → 假设标注）
    → 6 步分析流程（泳道 → 用例 → 状态机 → 字段溯源 → 时序 → 边界）
      → 输出草案文档
```

---

## 收敛三阶段（参考 idea-refine Phase 2）

### 阶段一：聚类页面结构（Cluster → Pages）

**Goal:** 将共鸣的功能变体聚类为页面组。

用户对变体的反馈后，聚类为 2-3 个页面组方向：

```markdown
### 方向 A：用户自助型售后

核心设计：用户在小程序端自助完成申请，后台仅做最终审核

页面组：
- 小程序端：申请页、进度页、详情页（共 3 页）
- 后台端：审核列表、审核详情（共 2 页）

选中的变体：变体 5（仅退款）+ 变体 3（用户自助）

为何选择：最小化页面数量，聚焦核心闭环

### 方向 B：客服代操作型售后

核心设计：客服在后台代用户发起申请，小程序端仅查看进度

页面组：
- 小程序端：进度页、详情页（共 2 页）
- 后台端：代申请页、审核列表、审核详情（共 3 页）

选中的变体：变体 3（客服代操作）

为何选择：用户端最简化，适合低频场景
```

**聚类规则：**

- 每个方向要明确"为何选择"
- 页面数量要具体（总数 ≤ 10 页为宜）
- 小程序端 + 后台端都要覆盖

---

### 阶段二：压力测试（Stress-test）

**Goal:** 每个页面/功能点需评估三维度。

参考 idea-refine refinement-criteria.md 的评估维度：

| 维度 | 评估问题 | 判断标准 |
|------|----------|----------|
| **User Value** | 这是止痛药还是维生素？ | Painkiller（解决强烈痛点）优先 |
| **Feasibility** | 技术实现难度？依赖哪些现有组件？ | 能复用现有组件 > 需新建组件 |
| **Differentiation** | 与现有处理方式有何不同？ | 必须比现状更好，否则不建 |

**压力测试格式：**

```markdown
### 页面：售后申请页

| 维度 | 评估 | 结论 |
|------|------|------|
| User Value | 用户投诉"找不到退款入口"，痛点强烈 | Painkiller ✓ |
| Feasibility | 复用订单详情页结构，新增表单组件 | Medium，可复用 |
| Differentiation | 现状：客服电话申请，耗时 10 分钟 | 新流程：自助 2 分钟，显著提升 |

关键风险：用户可能不理解"仅退款"和"退货退款"的区别

### 页面：审核列表页

| 维度 | 评估 | 结论 |
|------|------|------|
| User Value | 运营痛点：投诉积压，处理效率低 | Painkiller ✓ |
| Feasibility | 复用后台订单列表结构 | Low，直接复用 |
| Differentiation | 现状：无独立售后入口，混杂在订单中 | 新流程：独立入口，状态筛选 |

关键风险：审核时效要求需明确（24h/48h/72h）
```

**诚实评估原则：**

- 如果功能点是 Vitamin（锦上添花），明确标注
- 如果可行性 Low 且无复用组件，建议排除
- 如果与现状差异不大，质疑是否值得建

---

### 阶段三：显性假设（Surface Assumptions）

**Goal:** 每个页面标注三类假设，这是最重要的一步。

参考 idea-refine 的假设审计方法：

**假设分类：**

| 假设类型 | 标记 | 含义 | 验证方式 |
|----------|------|------|----------|
| **Must Be True** | `[必验]` | 如果错误，整个流程失败 | 原型验证或用户访谈 |
| **Could Kill** | `[风险]` | 可能导致功能失败或体验差 | 原型验证或数据佐证 |
| **Ignoring** | `[暂略]` | 当前版本有意排除 | 下版本决策 |

**假设标注格式：**

```markdown
### 售后申请页假设清单

| 假设 | 类型 | 验证方式 | 如果错误的应对 |
|------|------|----------|----------------|
| 用户愿意在小程序端上传凭证图片 | [必验] | 5 用户访谈 | 提供客服代上传入口 |
| 审核时效超 48h 用户会流失 | [风险] | 抽样历史投诉数据 | 自动催办提醒 |
| 暂不考虑跨仓库退货的物流调度 | [暂略] | — | V0.2 决策 |
| 用户能区分"仅退款"和"退货退款" | [必验] | 原型可用性测试 | 增加 tooltip 说明 |
| 运营能在 10 分钟内完成单条审核 | [风险] | 后台原型测试 | 简化审核字段 |

### 审核列表页假设清单

| 假设 | 类型 | 验证方式 | 如果错误的应对 |
|------|------|----------|----------------|
| 运营每天处理量 ≤ 100 条 | [风险] | 运营访谈 | 增加批量操作 |
| 审核操作需要备注字段 | [必验] | 运营访谈 | 决定是否增加备注输入 |
```

## Not Doing 清单（聚焦的核心）

**Goal:** 明确排除的功能，聚焦是第一原则。

参考 idea-refine："The 'Not Doing' list is arguably the most valuable part."

**Not Doing 格式：**

```markdown
### Not Doing 清单

| 排除项 | 排除理由 | 可能何时加入 |
|--------|----------|--------------|
| 售后聊天功能 | 非核心，可用客服系统替代 | V0.3（如有独立客服模块） |
| 自动退款审批 | 风险高，需人工审核兜底 | V0.2（基于规则引擎） |
| 退货物流跟踪 | 暂不对接物流系统 | V0.2（对接物流 API） |
| 用户撤回申请 | 简化流程，减少状态 | V0.2（按用户反馈决定） |
| 评价关联售后 | 评价和售后流程独立 | V0.3（联动设计） |
```

**排除原则：**

- 每项必须说明"为何排除"
- 区分"暂不做"和"永不做"
- Not Doing 项目数应 ≥ Doing 项目数的 50%

## 6 步分析流程（落地套路）

收敛阶段完成后，按以下 6 步顺序进行结构化分析。每步的输出直接写入草案文档的对应章节。

### 步骤一：画角色泳道图

**目标：** 理清有哪些角色，各做什么。

**输入：** 收敛阶段的页面组（方向 A/B/C）
**输出：** 写入文档 `角色泳道图` 章节

**方法：** 参考 [rules/business-analysis.md](business-analysis.md) 方法一

| 分析项 | 说明 |
|--------|------|
| 识别角色 | 列出所有参与方（用户、运营、前端、后端、第三方） |
| 按时间排列 | 按用户操作路径从入口到出口 |
| 检查遗漏 | 操作无反馈、异常无提示的空白格 |

**输出格式：**

```markdown
## {N}、角色泳道图 `V{版本号}-草案`

> `[草案]` 角色划分和交互关系待原型验证

| 阶段 | 用户 | 前端 | 后端服务 | 数据库 |
|------|------|------|----------|--------|
| {阶段1} | {操作} | {调用} | {处理} | {读写} |
```

### 步骤二：拆功能用例

**目标：** 逐功能展开正常和异常场景。

**输入：** 角色泳道图中的用户操作
**输出：** 写入每页的 `功能用例表` 章节

**方法：** 十字法（参考 [rules/use-case-generation.md](use-case-generation.md) 十字法提取框架）

| 横向追问 | 映射到列 |
|----------|----------|
| Who（角色） | 场景列 |
| When/Where（前置条件） | 前置条件列 |
| What（操作） | 操作步骤列 |
| Result（正常结果） | 预期结果列 |

| 纵向追问 | 映射到列 |
|----------|----------|
| 边界条件（为空/超限/重复/并发） | 异常分支列 |
| 失败后用户看到什么 | 异常分支列 |

**输出格式：**

```markdown
#### {X}.{Y}.1 功能用例表

| # | 场景 | 前置条件 | 操作步骤 | 预期结果 | 异常分支 |
|---|------|----------|----------|----------|----------|
| 1 | {场景} | {前置条件} | {操作} | {预期结果} `[草案]` | {异常} `[草案]` |
```

### 步骤三：画状态机

**目标：** 有状态的对象单独建模型。

**输入：** 功能用例中涉及状态变更的操作
**输出：** 写入文档 `状态流转` 章节

**方法：** 参考 [rules/business-analysis.md](business-analysis.md) 方法二

**触发条件：** 当实体有 ≥ 3 个状态时执行此步骤。

| 分析项 | 说明 |
|--------|------|
| 列出所有状态 | 含初始和终态 |
| 画跳转路径 | 每个状态可跳到哪些状态 |
| 标注触发条件 | 什么事件触发状态变化 |
| 标注守卫条件 | 跳转还需满足什么条件 |
| 覆盖超时和异常 | 每个等待状态的超时处理 |

### 步骤四：做字段溯源

**目标：** 每个关键页面的字段过一遍。

**输入：** 页面中的所有字段
**输出：** 写入每页的 `关键字段数据来源` 章节

**方法：** 数据溯源三问（参考 [rules/business-analysis.md](business-analysis.md) 方法三）

| 三问 | 含义 |
|------|------|
| 从哪来 | 字段的值从何产生 |
| 往哪去 | 提交到哪个接口 / 写入哪张表 |
| 变了怎么办 | 来源变更时是否同步 |

**输出格式：**

```markdown
#### {X}.{Y}.2 关键字段数据来源

| 字段 | 从哪来 | 往哪去 | 变了怎么办 |
|------|--------|--------|------------|
| {字段名} | {来源} | {目的地} `[待确认]` | {同步策略} `[草案]` |
```

### 步骤五：串系统时序

**目标：** 跨系统的请求链路走通。

**输入：** 角色泳道图 + 状态机 + 字段溯源
**输出：** 写入文档 `系统交互时序` 章节

**方法：** 参考 [rules/business-analysis.md](business-analysis.md) 方法五

| 分析项 | 说明 |
|--------|------|
| 列出跨系统操作 | 前端↔后端↔数据库↔第三方 |
| 标注同步/异步 | 决定是否需要重试和补偿 |
| 定义失败策略 | 重试次数 + 告警 + 人工补偿 |

**输出格式：**

```markdown
## {N}、系统交互时序 `V{版本号}-草案`

| # | 触发操作 | 请求链路 | 同步/异步 | 失败策略 |
|---|----------|----------|-----------|----------|
| 1 | {操作} `[草案]` | {链路} | {同步/异步} | {策略} `[草案]` |
```

### 步骤六：补充边界条件

**目标：** 覆盖并发、幂等、数据一致性。

**输入：** 前 5 步分析中识别的边界风险点
**输出：** 写入文档 `边界条件清单` 章节

**方法：** 边界类型分类

| 边界类型 | 典型场景 | 处理策略示例 |
|----------|----------|-------------|
| 并发 | 多人同时抢购同一SKU | 乐观锁 + 库存预扣 |
| 幂等 | 重复点击提交按钮 | 按钮禁用 + 请求去重 |
| 数据一致性 | 支付成功但订单状态未更新 | 定时对账 + 补偿 |
| 超限 | 数量超过库存 / 金额超限 | 前端限制 + 后端校验 |
| 时序 | 操作顺序颠倒（先发货再付款） | 状态机守卫条件 |

**输出格式：**

```markdown
## {N}、边界条件清单 `V{版本号}-草案`

| # | 边界类型 | 场景 | 处理策略 |
|---|----------|------|----------|
| 1 | {类型} | {场景} `[草案]` | {策略} `[草案]` |
```

## 6 步流程检查清单

进入文档输出前确认：

- [ ] 角色泳道图已画（角色 ≥ 2 个，阶段覆盖完整流程）
- [ ] 每个页面功能用例表已完成（6 列完整，异常分支非空或标注"—"）
- [ ] 有状态的实体已建模状态机（状态 ≥ 3 个时必做）
- [ ] 每个页面字段溯源已完成（三问缺一不可）
- [ ] 跨系统交互已画出时序（标注同步/异步）
- [ ] 边界条件已补充（至少覆盖并发、幂等、一致性）

**Goal:** 将功能大纲映射到业务逻辑清单三表结构。

### 映射规则

| 功能大纲项 | 映射到清单章节 | 状态标记 |
|------------|----------------|----------|
| 用户动作 | 功能用例表"操作步骤"列 | `[草案]` |
| 前置条件 | 功能用例表"前置条件"列 | `[草案]` |
| 设计意图 | 功能用例表"预期结果"列 | `[草案]` |
| 异常路径 | 功能用例表"异常分支"列 | `[草案]` |
| 数据需求（从哪来） | 关键字段数据来源"从哪来"列 | `[待确认]`（来源未定） |
| 数据去向（往哪去） | 关键字段数据来源"往哪去"列 | `[待确认]` |
| 数据变更策略 | 关键字段数据来源"变了怎么办"列 | `[草案]` |
| 待确认规则 | 业务逻辑增强表 | `[待确认]` |
| 假设清单 | 业务逻辑增强表 | `[必验]`/`[风险]`/`[暂略]` |

| 角色交互 | 角色泳道图章节 | `[草案]` |
| 状态变更 | 状态流转章节 | `[草案]` |
| 系统调用 | 系统交互时序章节 | `[草案]` |
| 边界风险 | 边界条件清单章节 | `[草案]` |

### 功能大纲 → 功能用例转换


## 草案输出格式

### 文档头部

```markdown
# {项目名} - 业务逻辑清单 V{版本号}-草案

> **草案说明**：本文档基于需求描述生成，功能用例为设计意图，
> 预期结果标注 `[草案]`，待原型验证后替换为实现描述。
>
> **假设验证**：标注 `[必验]`/`[风险]`/`[暂略]` 的假设项需在原型阶段验证。
>
> **HMW 问题陈述**：How might we {问题重述}？
>
> **成功标准**：{澄清问题阶段定义的验收指标}

---

## Not Doing 清单

| 排除项 | 排除理由 | 可能何时加入 |
|--------|----------|--------------|
| ... | ... | ... |

---
```

```markdown

>

> - {假设1} `[必验]` — 验证方式：{...}
> - {假设2} `[风险]` — 验证方式：{...}

#### 1.1.1 功能用例表

| # | 场景 | 前置条件 | 操作步骤 | 预期结果 | 异常分支 |
|---|------|----------|----------|----------|----------|
| 1 | ... | ... | ... | ... `[草案]` | ... `[草案]` |

#### 1.1.2 关键字段数据来源

| 字段 | 从哪来 | 往哪去 | 变了怎么办 |
|------|--------|--------|------------|
| ... | ... | ... `[待确认]` | ... `[草案]` |

#### 1.1.3 业务逻辑增强
```

## Anti-patterns（反模式）

| 反模式 | 正确做法 |
|--------|----------|
| 不标注假设 | 每个页面必须有假设清单，至少 2 条 |
| 不做 Not Doing 清单 | Not Doing 项目数 ≥ Doing 的 50% |
| 假设类型不区分 | 必须用 [必验]/[风险]/[暂略] 区分 |
| 压力测试一味说好 | 诚实评估，Vitamin 功能明确标注 |
| 草案预期结果写得太模糊 | 具体到交互元素名称、跳转页面名 |
| 页面数量过多 | 总页面数 ≤ 10，聚焦闭环 |
| 跳过角色泳道直接写用例 | 先理清角色职责，再展开用例（步骤一→步骤二） |
| 用例表只写正常路径 | 十字法纵向追问：每条正常用例必须有异常分支 |
| 数据来源只写"从哪来" | 三问缺一不可，尤其是"变了怎么办" |
| 状态机只画正常流转 | 每个状态必须覆盖超时和异常路径 |
| 系统时序不标注同步/异步 | 每条链路必须标注，决定失败策略 |
| 边界条件只写"需处理" | 必须写明具体处理策略 |
