# 文档结构规则 V2.0

定义业务逻辑清单的章节编排规则：标题层级、编号规则、表格结构、全局章节。融合十字法和6步分析方法论。

---

## 章节结构总览

```
业务逻辑清单 V{版本号}
├── 文档标题 + 版本概述（引用块）
├── 一、{模块名} `V{版本号}`
│   ├── 1.1 {页面名}（filename.html）
│   │   ├── 截图                   [可选，默认关闭，用户要求时加]
│   │   ├── 1.1.1 功能用例表       [必选，6列]
│   │   ├── 1.1.2 关键字段数据来源   [必选，4列溯源三问]
│   │   └── 1.1.3 业务逻辑增强       [可选]
│   └── 1.2 {关系表/跳转表}         [按需]
├── 二、{模块名} `V{版本号}`
│   └── ...
├── {N}、页面导航 `V{版本号}`       [必选]
│   ├── {N}.1 跳转关系
│   ├── {N}.2 返回导航
│   └── {N}.3 登录拦截
├── {N+1}、状态流转 `V{版本号}`     [可选]
├── {N+2}、角色泳道图 `V{版本号}`   [Draft必选/Extract按需]
├── {N+3}、核心单据流转 `V{版本号}` [有核心实体时必选]
├── {N+4}、系统交互时序 `V{版本号}` [跨系统时必选]
└── {N+5}、边界条件清单 `V{版本号}` [必选]
```

## 标题层级规则

### 一级标题 `#` — 文档标题

每个文档仅一个，格式：`# {项目名} - 业务逻辑清单 V{版本号}（{版本描述}）`

示例：
```
# 苏银豆商城 - 业务逻辑清单 V0.3（下单与评价流程）
```

### 二级标题 `##` — 模块章节

使用中文数字序号，格式：`## {中文数字}、{模块名} \`V{版本号}\``

示例：
```
## 一、商品详情 `V0.3`
## 二、购物车 `V0.3`
## 三、下单与支付 `V0.3`
```

规则：
- 每个二级标题后用 `---` 分隔
- 按用户操作路径排序（浏览→加购→下单→支付→订单→评价）

### 三级标题 `###` — 页面小节

使用阿拉伯数字编号，格式：`### {X}.{Y} {页面名}（{filename.html}）`

示例：
```
### 1.1 商品详情页（product_detail.html）
### 3.1 确认订单（order.html）
```

### 四级标题 `####` — 子小节

使用三级编号，格式：`#### {X}.{Y}.{Z} {表格类型名}`

固定编号含义：
- `.1` = 功能用例表
- `.2` = 关键字段数据来源
- `.3` = 业务逻辑增强

示例：
```
#### 1.1.1 功能用例表
#### 1.1.2 关键字段数据来源
#### 1.1.3 业务逻辑增强
```

## 版本概述

文档开头使用引用块 `>` 写版本概述，包含：

| 内容 | 说明 |
|------|------|
| 版本范围 | 本次覆盖哪些模块和页面 |
| 页面数量 | 涉及的页面总数 |
| 数据来源说明 | 商品数据、用户数据等来源定义 |
| 统计口径 | 销量、好评率、折扣等指标的计算公式 |
| 测试说明 | 如 `Auth.resetAccounts()` 重置提示 |

## 三表结构

### 功能用例表（必选，6列）

每个页面必须有。基于十字法（横向角色×场景，纵向正常×异常）。

```markdown
| # | 场景 | 前置条件 | 操作步骤 | 预期结果 | 异常分支 |
|---|------|----------|----------|----------|----------|
| 1 | ... | ... | ... | ... | ... |
```

列定义：
- **#**：页内顺序编号，每个表从 1 开始
- **场景**：简短名词短语（2-6字），描述在什么情境下
- **前置条件**：触发该用例前必须满足的条件（角色×场景的 When/Where 维度）
- **操作步骤**：用户的具体动作，包含元素名称
- **预期结果**：具体到文案、颜色（附十六进制值）、动画、跳转目标（附页面文件名）
- **异常分支**：该操作可能触发的异常路径及处理（正常×异常维度），格式：`异常条件 → 处理方式`

### 关键字段数据来源（必选，4列溯源三问）

每个页面必须有。基于数据溯源三问方法。

**概念说明：** "关键字段数据来源"不是指来源于某个代码文件，而是指**页面字段在业务层面的数据来源**，即该字段的值从哪个业务系统或页面环节产生。

```markdown
| 字段 | 从哪来 | 往哪去 | 变了怎么办 |
|------|--------|--------|------------|
| ... | ... | ... | ... |
```

列定义：
- **字段**：字段名 + 括号列举子字段（如 `商品信息（图片/名称）`）
- **从哪来**：字段的值从何产生（用户输入/接口返回/本地下拉/前端计算/上一页传入/系统生成）
- **往哪去**：提交到哪个接口/写入哪张表/触发哪个流程/仅展示不提交
- **变了怎么办**：来源数据变更时的处理策略（实时同步/手动刷新/不影响已提交/触发通知）

## 数据来源分类

关键字段数据来源表的"从哪来"列使用以下通用分类：

| 分类 | 说明 | 示例写法 |
|------|------|----------|
| 系统后台 | 后端系统提供的业务数据 | `{系统名}后台 {模块名}`（如：运营后台用户管理、CRM系统客户数据） |
| 用户操作 | 用户交互产生的数据 | `用户操作`（如：勾选状态、输入内容、选择项） |
| 前端计算 | 前端实时计算得出 | `前端实时计算`（如：合计金额、字数统计、倒计时） |
| 上一页面传入 | 从上游页面携带的数据 | `上一页面传入`（如：商品ID、订单ID、编辑对象） |
| 本地存储 | 浏览器持久化的数据 | `本地存储`（如：localStorage、sessionStorage） |
| 第三方服务 | 外部API提供的数据 | `{服务商名}`（如：微信SDK、地图API、支付网关） |
| 配置文件 | 系统配置项 | `系统配置`（如：超时时间、阈值、开关状态） |
| 系统生成 | 后端服务自动生成 | `系统生成`（如：订单号、流水号、时间戳） |

**不确定的数据来源**：留白并附建议答案，格式：
```
{系统名}后台{模块名}（建议：{具体模块名}）
```

### 业务逻辑增强（可选）

仅在存在跨页面或需要业务确认的逻辑问题时生成。

```markdown
| # | 问题 | 确认结果 |
|---|------|----------|
| 1 | ... | ... |
```

列定义：
- **#**：页内顺序编号
- **问题**：以问号结尾的业务逻辑疑问
- **确认结果**：已确认的回答；未确认的留空并附建议

**不确定的业务问题**：确认结果列留白并附建议答案，格式：
```
（建议：{建议的确认结果}）
```

## 全局章节

### 跳转关系

```markdown
| # | 页面A | 触发条件 | 页面B |
|---|-------|----------|-------|
| 1 | 购物车 | 点击"结算" | 确认订单 |
```

覆盖版本范围内所有页面的跳转关系，编号全局递增。

### 返回导航

一行说明：
```
所有子页面返回按钮统一使用 `history.back()` 返回来源页面。
```

### 登录拦截

```markdown
| 类型 | 页面 |
|------|------|
| 受保护 | {需登录才能访问的页面列表} |
| 公开 | {无需登录即可访问的页面列表} |
```

后附引用块说明拦截机制（如 `auth-guard.js` 或自定义拦截逻辑）。

### 状态流转（可选）

仅当涉及状态机（如订单状态、审批状态、任务状态）时生成。

```markdown
| 当前状态 | 可执行操作 | 目标状态 |
|----------|-----------|----------|
| {状态A} | {操作} | {状态B} |
```

可配合 mermaid stateDiagram-v2 展示完整流转路径。

### 角色泳道图（Draft 必选，Extract 按需）

多角色协作流程的交互矩阵。详细规则见 [rules/business-analysis.md](business-analysis.md) 方法一。

```markdown
| 阶段 | 用户 | 前端 | 后端服务 | 数据库 |
|------|------|------|----------|--------|
| {阶段1} | {操作} | {调用} | {处理} | {读写} |
```

**生成条件：** 流程涉及 ≥ 2 个角色/系统时必须生成。

### 核心单据流转（有核心实体时必选）

以核心实体为单位的生命周期描述。详细规则见 [rules/business-analysis.md](business-analysis.md) 方法四。

```markdown
### {实体名}生命周期

创建 → 提交 → 审核 → 生效 → 变更 → 作废

| 阶段 | 参与字段 | 系统响应 |
|------|----------|----------|
| 创建 | {字段列表} | {创建逻辑} |
```

**生成条件：** 涉及订单、审批、工单等有生命周期的核心实体时必须生成。

### 系统交互时序（跨系统时必选）

跨系统请求链路的时序描述。详细规则见 [rules/business-analysis.md](business-analysis.md) 方法五。

```markdown
| # | 触发操作 | 请求链路 | 同步/异步 | 失败策略 |
|---|----------|----------|-----------|----------|
| 1 | {操作} | {链路} | 同步 | {策略} |
```

**生成条件：** 涉及前端↔后端↔第三方等跨系统交互时必须生成。

### 边界条件清单（必选）

覆盖并发、幂等、数据一致性等边界场景。

```markdown
| # | 边界类型 | 场景 | 处理策略 |
|---|----------|------|----------|
| 1 | 并发 | {场景} | {策略} |
| 2 | 幂等 | {场景} | {策略} |
| 3 | 数据一致性 | {场景} | {策略} |
```

**边界类型枚举：** 并发、幂等、数据一致性、超限、时序。

## 编号连续性要求

1. H2 中文数字不跳号：一、二、三、四、五、六、七、八、九
2. H3 在同一 H2 下不跳号：1.1、1.2、1.3
3. H4 在同一 H3 下不跳号：1.1.1、1.1.2、1.1.3
4. 功能用例表和业务逻辑增强表内 `#` 列不跳号
5. 跳转关系表 `#` 列全局递增不跳号
6. 系统交互时序表 `#` 列全局递增不跳号
7. 边界条件清单表 `#` 列全局递增不跳号

## 模块编排原则

按**用户操作路径**排序，遵循以下通用原则：

### 编排规则

1. **流程概述优先** — 若版本涉及完整业务流程（如订单流程、审批流程），可将流程图和状态流转作为首章概述
2. **按操作顺序排列** — 从用户进入系统到完成目标操作的路径顺序
3. **入口页面优先** — 用户最先访问的页面排在前面
4. **详情页在列表页后** — 列表→详情的层级关系
5. **全局章节固定末尾** — 页面导航、状态流转（若非首章）放在文档末尾

### 示例模板

**电商下单流程**：
```
一、订单流转（流程概述） → 二、商品详情 → 三、购物车 → 四、下单支付 → 五、订单管理 → 六、页面导航
```

**内容管理流程**：
```
一、内容流转（流程概述） → 二、内容列表 → 三、内容详情 → 四、内容编辑 → 五、页面导航
```

**审批流程**：
```
一、审批流转（流程概述） → 二、待审批列表 → 三、审批详情 → 四、审批操作 → 五、页面导航
```

**用户中心**：
```
一、个人信息 → 二、账号设置 → 三、安全设置 → 四、页面导航
```

## 常见错误

| 错误 | 正确做法 |
|------|----------|
| 按文件名字母排序 | 按用户操作路径排序 |
| 所有页面都有业务逻辑增强表 | 仅在有需要确认的业务问题时生成 |
| H4 不带编号 | 所有 H4 必须带 X.Y.Z 编号 |
| 截图使用绝对路径 | 使用相对路径 `./screenshots/xxx.png` |
| 跳转表仅列部分跳转 | 必须覆盖版本范围内所有页面的全部跳转 |
| 功能用例表只用4列 | 必须使用6列（含前置条件和异常分支） |
| 数据来源只写"从哪来" | 必须使用4列溯源三问格式 |
| 缺少角色泳道图 | Draft模式涉及 ≥ 2 个角色时必须生成 |
| 缺少边界条件清单 | 所有版本必须包含边界条件清单章节 |
