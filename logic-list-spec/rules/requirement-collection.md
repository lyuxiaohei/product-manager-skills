# 需求收集规则 V1.0

深度融合 idea-refine 方法论，定义 Draft 模式的需求收集流程。

---

## 核心原则（继承 idea-refine）

1. **Simplicity is the ultimate sophistication** — 推向最简单的页面结构，仍解决真实问题
2. **Start with the user experience, work backwards** — 先设计用户体验，再推导后台功能
3. **Say no to 1,000 things** — Not Doing 清单是核心产出，聚焦比功能堆砌更重要
4. **Challenge every assumption** — 每条"常规做法"都需质疑
5. **The parts you can't see should be as beautiful** — 后台流程虽用户不可见，设计需同样精细

---

## 需求收集三阶段

### 阶段一：问题重述（How Might We）

**Goal:** 强制先提炼问题陈述，避免需求讨论偏离。

将用户的原始需求重述为 "How Might We" 格式：

```
原始需求："我要做一个售后流程，用户可以在小程序申请退款，后台审核"

HMW重述："How might we 让用户在小程序端便捷发起售后申请，
          同时运营在后台高效审核处理，实现完整的退换货闭环？"
```

**重述要点：**

| 要点 | 说明 |
|------|------|
| 双端视角 | 小程序端（用户）+ 后台端（运营） |
| 核心动词 | 发起、审核、处理、闭环 |
| 价值目标 | "便捷"、"高效"、"完整闭环" |

**输出：** 一段 HMW 问题陈述，写入草案文档头部。

---

### 阶段二：澄清问题（Sharpening Questions）

**Goal:** 限制 5 个问题以内，聚焦核心维度。

使用 `AskUserQuestion` 工具收集以下 5 个维度：

| # | 问题 | 目的 |
|---|------|------|
| 1 | **用户是谁？** | 明确小程序端用户画像（新用户/老用户/高频用户） |
| 2 | **成功标准是什么？** | 定义验收指标（完成率、时效、满意度、投诉率） |
| 3 | **技术约束是什么？** | 现有架构限制、时间窗口、依赖系统 |
| 4 | **尝试过什么？** | 现有系统的售后能力、痛点、遗留问题 |
| 5 | **为什么现在做？** | 业务驱动（投诉上升/竞品对标/合规要求） |

**不进入下一阶段，直到：**

- [ ] 用户画像明确
- [ ] 成功标准量化或定性定义
- [ ] 技术约束已识别

---

### 阶段三：变体生成（Idea Variations）

**Goal:** 生成 5-8 个功能方向变体，质量优于数量。

复用 idea-refine 的 7 种视角，但聚焦"页面/功能维度"：

| 视角 | 转换为页面设计 | 应用示例 |
|------|----------------|----------|
| **Inversion** | 逆向流程设计 | 用户端申请 vs 商家端主动退款（仅退款无需申请） |
| **Constraint removal** | 无约束理想版本 | 无审核流程 vs 三级审批流程 |
| **Audience shift** | 目标用户置换 | 面向用户自助 vs 面向客服代操作 |
| **Combination** | 功能联动设计 | 售后+投诉+评价联动 vs 售后独立模块 |
| **Simplification** | 10x 简化版本 | 仅退款（无退货） vs 退换货一体化 |
| **10x version** | 大规模理想版本 | 全自动化 AI 审核+智能退款 vs 人工审核兜底 |
| **Expert lens** | 专家视角洞察 | 电商售后专家如何设计状态机（状态数、流转条件） |

**变体编写格式：**

```markdown
### 变体 1：{视角名称} — {方向标题}

核心设计：{一句话描述该方向}

页面组：
- 小程序端：{页面列表}
- 后台端：{页面列表}

核心假设：
- {假设1}（可能错误的假设）
- {假设2}

为何值得探索：{2-3句话}
```

**生成规则：**

- 不超过 8 个变体（质量优于数量）
- 每个变体必须说明"为何值得探索"
- 变体之间要明显不同，不是同一方向的不同版本

---

### 阶段四：上下文扫描（Codebase-Aware）

**Goal:** 扫描现有代码库，识别可复用组件和约束。

参考 idea-refine 的 codebase 扫描要求，使用 `Glob`、`Grep`、`Read`：

| 扫描维度 | 搜索模式 | 识别内容 |
|----------|----------|----------|
| 现有订单流程 | `order*.html` | 订单详情页、订单状态标签、订单操作按钮 |
| 状态流转定义 | `status.*\|state.*` | 状态枚举、状态颜色、状态文案 |
| 表单提交模式 | `submit.*\|form.*` | 表单校验、提交按钮、成功/失败提示 |
| 弹窗组件 | `popup.*\|modal.*\|overlay.*` | 弹窗结构、确认按钮样式 |
| 后台审批模式 | `review.*\|approve.*`（如有） | 审批列表、审批详情、审批按钮 |
| 图片上传 | `upload.*\|image.*` | 上传组件、图片预览、数量限制 |

**扫描结果记录：**

```markdown
### 可复用组件

| 组件 | 来源文件 | 复用方式 |
|------|----------|----------|
| 状态标签 | `order_detail.html` | 售后状态复用订单状态标签样式 |
| 确认弹窗 | `payment.html` | 提交申请弹窗复用支付确认弹窗结构 |
| 图片上传 | `review.html` | 凭证上传复用评价页图片上传组件 |

### 现有约束

| 约束 | 来源 | 影响 |
|------|------|------|
| 订单状态不可逆 | `order_status` 定义 | 售后状态需独立状态机，不修改订单状态 |
| 仅支持 3 张图片 | `review.html` 限制 | 凭证上传同样限制 3 张 |
```

## 需求收集检查清单

进入页面收敛阶段前，确认以下项：

- [ ] HMW 问题陈述已提炼
- [ ] 5 个澄清问题已回答（至少用户画像+成功标准）
- [ ] 5-8 个功能变体已生成，每个有明确理由
- [ ] 上下文扫描已完成，可复用组件已识别
- [ ] 现有约束已记录

---

## Anti-patterns（反模式）

| 反模式 | 正确做法 |
|--------|----------|
| 直接进入页面设计 | 先完成 HMW 重述和澄清问题 |
| 生成 20+ 个变体 | 限制 5-8 个，质量优于数量 |
| 变体之间过于相似 | 每个变体要明显不同方向 |
| 跳过 codebase 扫描 | 必须扫描现有组件和约束 |
| 不记录假设 | 每个变体必须标注核心假设 |
| 不说"不做什么" | Not Doing 清单是聚焦的关键 |

---

## Tone（语气风格）

继承 idea-refine 的语气：

- Direct, thoughtful, slightly provocative
- 是"思考伙伴"而非"流程执行者"
- 推向一步更深："那很有趣，但如果..."
- 诚实而非一味支持：指出弱项假设
