# 模式识别规则 V2.0

定义如何识别用户输入的模式类型（口述模式 vs 草案模式）。

V2.0 变更：表格结构检测适配 logic-list-spec V0.22 的 6 列用例表和 4 列溯源三问字段表。

---

## 模式类型

| 模式 | 输入特征 | 处理流程 |
|------|----------|----------|
| **草案模式** | 结构化草案文档 | 草案解析 → 用例映射 → 字段映射 → 状态映射 → HTML生成 |
| **口述模式** | 自然语言描述 | 关键词提取 → 页面推断 → 组件选择 → HTML生成 |

---

## 检测优先级

**草案模式优先**：满足草案条件则进入草案模式，否则默认口述模式。

---

## 草案模式触发条件

满足以下**任一条件**即判定为草案模式：

### 1. 结构化标记检测

| 标记类型 | 检测规则 | 权重 |
|----------|----------|------|
| 状态标记 | 包含 `[草案]`、`[待原型]`、`[待确认]`、`[必验]`、`[风险]` | 高 |
| HMW关键字 | 包含 `How Might We` 或 `HMW` 问题陈述格式 | 高 |
| Not Doing标识 | 包含 `Not Doing` 清单章节或标识 | 高 |

### 2. 表格结构检测

| 表格类型 | 检测规则 | 权重 |
|----------|----------|------|
| 功能用例表 | 包含 `功能用例表` 标题 + `场景/操作步骤/预期结果` 表头（V0.22 六列含前置条件/异常分支） | 高 |
| 状态流转表 | 包含 `状态流转` 标题 + `当前状态/目标状态` 表头 | 中 |
| 关键字段数据来源 | 包含 `关键字段数据来源` 标题 + `字段/从哪来` 表头（V0.22 四列含往哪去/变了怎么办） | 中 |
| 跳转关系表 | 包含 `跳转关系` 标题 + `页面A/页面B` 表头 | 中 |

**兼容性：** V0.22 六列和旧版四列用例表均可触发草案模式检测（场景/操作/预期结果 三个核心表头仍存在）。

### 3. 章节结构检测

| 章节类型 | 检测规则 | 权重 |
|----------|----------|------|
| 页面定位 | 包含 `页面定位` 章节或引用块 | 中 |
| 假设清单 | 包含 `假设清单` 章节 | 中 |
| 压力测试结论 | 包含 `User Value/Feasibility/Differentiation` 三维度 | 中 |

### 4. 显式声明

| 声明类型 | 检测规则 | 权重 |
|----------|----------|------|
| 用户明确声明 | 用户提到"草案"、"业务逻辑清单"、"logic-list-spec输出" | 高 |
| 文件引用 | 用户引用 `.md` 文件路径 | 中 |

---

## 口述模式判定

不满足草案模式触发条件的输入，判定为口述模式。

**口述模式特征：**
- 纯文本描述，无结构化标记
- 用户自由描述页面需求
- 无表格结构
- 无章节结构

---

## 检测算法伪代码

```python
def detect_mode(input_text):
    # 高权重检测（草案模式）
    if contains_draft_markers(input_text):  # [草案]、[待原型]、HMW、Not Doing
        return "draft"
    
    if contains_explicit_declaration(input_text):  # 用户明确声明
        return "draft"
    
    if contains_table_structure(input_text):  # 功能用例表、状态流转表
        return "draft"
    
    # 中权重检测（草案模式）
    draft_score = 0
    if contains_chapter_structure(input_text):  # 页面定位、假设清单
        draft_score += 1
    if contains_status_flow(input_text):  # 状态流转定义
        draft_score += 1
    
    if draft_score >= 2:
        return "draft"
    
    # 默认口述模式
    return "oral"
```

## 检测示例

### 草案模式示例

```markdown
# 业务逻辑清单 V0.3-草案

> **HMW 问题陈述**：How might we 让用户便捷发起售后申请？

## Not Doing 清单

| 排除项 | 排除理由 |
|--------|----------|

### 1.1 申请页 `[待原型]`

> **页面定位**：用户填写售后申请

#### 1.1.1 功能用例表

| # | 场景 | 前置条件 | 操作步骤 | 预期结果 | 异常分支 |
|---|------|----------|----------|----------|----------|
| 1 | 选择类型 | 页面加载完成 | 点击Tab | 选中项高亮 `[草案]` | — |
```

**检测结果：草案模式**
- 包含 `[草案]` 状态标记 ✓
- 包含 HMW 问题陈述 ✓
- 包含 Not Doing 清单 ✓
- 包含功能用例表 ✓

### 口述模式示例

```markdown
设计一个商品列表页面，运营人员可以查看所有商品，支持搜索、筛选、分页，点击可以查看详情
```

**检测结果：口述模式**
- 无状态标记
- 无表格结构
- 无章节结构
- 纯文本描述

## 混合模式处理

当输入同时包含口述描述和草案结构时：

1. **优先草案模式**：草案结构更精确，优先使用草案映射流程
2. **口述补充**：口述内容可作为草案的补充说明
3. **融合处理**：口述描述可覆盖草案中 `[待确认]` 的不确定项

---

## 错误处理

| 错误类型 | 处理方式 |
|----------|----------|
| 无法识别模式 | 默认口述模式，提示用户确认 |
| 草案结构不完整 | 进入草案模式，提示补充缺失章节 |
| 模式判断错误 | 允许用户手动指定模式 |
