# 文档结构规则 V6.0

定义版本PRD的章节编排规则：哪些章节必选、哪些可选、如何组织。

V6.0 变更：新增角色与架构、单据流转、接口时序、非功能需求4个章节，承接 logic-list-spec V0.22 的角色泳道图、核心单据流转、系统交互时序、边界条件清单。

---

## 章节结构总览

一份版本PRD由以下部分组成：

```
{版本号} PRD
├── 一、版本概述                    [必选]
├── 二、{模块名}（如：账号与登录）    [必选，按模块重复]
│   ├── 2.0 {模块前言}              [可选，模块级定义]
│   ├── 2.1 {页面名}
│   │   ├── 2.1.1 页面结构
│   │   ├── 2.1.2 操作流程
│   │   ├── 2.1.3 字段与交互
│   │   ├── 2.1.4 业务规则
│   │   └── 2.1.5 页面跳转
│   └── 2.2 {页面名} ...
├── {N}、数据模型                    [可选]
│   ├── {N}.1 数据结构
│   └── {N}.2 API 方法
├── {N+1}、角色与架构                [可选，≥2角色时有则必选]
├── {N+2}、单据流转                  [可选，有核心实体时必选]
├── {N+3}、接口时序                  [可选，跨系统交互时必选]
├── {N+4}、页面导航                  [必选]
│   ├── {N+4}.1 跳转关系
│   ├── {N+4}.2 返回导航
│   └── {N+4}.3 登录拦截
├── {N+5}、非功能需求                [必选]
└── 附录                             [可选]
```

---

## 必选章节

### 一、版本概述

放在文档最前面，回答以下问题：

| 内容 | 说明 |
|------|------|
| 版本范围 | 本次PRD覆盖哪些页面，列出清单 |
| 页面数量 | 本次涉及多少个页面 |
| 核心功能 | 用2-3句话概括本次版本的核心变更/新增 |
| 数据来源说明 | 商品数据、用户数据等的来源定义 |
| 统计口径 | 销量、好评率、折扣等指标的计算方式 |
| 示例账号 | 测试账号表（手机号、姓名、状态、用途） |

**注意**：版本概述不是目录。它是让读者在5分钟内理解"这个版本做了什么"的快速入口。

### 页面章节（二 ~ N-1）

按模块组织，每个模块下包含若干页面。页面章节使用 `templates/prd-page-section.md` 中的6节模板。

**模块前言**（可选）：部分模块需要在页面章节之前加一段模块级定义，如收藏维度说明（SKU vs SPU）、全局登录态管理等。使用 `{X}.0` 编号：

```
### 5.0 收藏维度与入口
（模块级定义，不绑定具体页面）

### 5.1 收藏页（favorites.html）
（页面章节）
```

### 页面导航（最后必选章节）

包含以下子节：

| 子节 | 编号 | 内容 |
|------|------|------|
| 跳转关系 | {N}.1 | 列出所有页面的入口和出口 |
| 返回导航 | {N}.2 | 说明返回按钮行为（通常统一使用 history.back()） |
| 登录拦截 | {N}.3 | 受保护页面列表、拦截逻辑、公开页面列表 |

## 可选章节

以下章节仅在确实存在相关内容时生成：

| 章节 | 生成条件 | 来源（logic-list-spec） | 示例 |
|------|----------|------------------------|------|
| 数据模型 | 有需要定义的数据结构 | 关键字段数据来源 + 核心单据流转 | 用户数据模型、商品数据模型 |
| 状态流转 | 有多状态的业务对象 | 状态流转表 | 订单状态、退款状态 |
| 角色与架构 | 流程涉及 ≥ 2 个角色/系统 | 角色泳道图 | 用户+运营+系统角色划分 |
| 单据流转 | 有核心业务实体（订单、审批等） | 核心单据流转 | 订单生命周期：创建→支付→发货→完成 |
| 接口时序 | 涉及跨系统交互 | 系统交互时序 | 小程序→网关→订单服务→DB |
| 全局业务规则 | 跨页面的通用规则 | 业务逻辑增强 | 登录态管理、收藏维度定义 |
| 附录 | 需要补充的参考资料 | — | API文档链接、术语表 |

### 角色与架构章节

当业务逻辑清单中包含"角色泳道图"章节时生成。

```markdown
## {N}、角色与架构

### {N}.1 角色定义

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| 用户（小程序端） | C端消费者 | 浏览、加购、下单、评价 |
| 运营（后台） | 商城运营人员 | 商品管理、订单管理、售后审核 |
| 系统 | 自动化服务 | 定时任务、通知推送、数据同步 |

### {N}.2 角色交互矩阵

（从业务逻辑清单的角色泳道图矩阵表转换）
```

### 单据流转章节

当业务逻辑清单中包含"核心单据流转"章节时生成。

```markdown
## {N}、单据流转

### {N}.1 {实体名}生命周期

（从业务逻辑清单的核心单据流转表转换）

创建 → 提交 → 审核 → 生效 → 变更 → 作废

| 阶段 | 参与字段 | 系统响应 | 前置条件 |
|------|----------|----------|----------|
| ... | ... | ... | ... |

### {N}.2 {实体名2}生命周期

（如涉及多个核心实体，逐一列出）
```

### 接口时序章节

当业务逻辑清单中包含"系统交互时序"章节时生成。

```markdown
## {N}、接口时序

### {N}.1 同步接口

| # | 操作 | 请求路径 | 请求参数 | 响应结构 | 异常码 |
|---|------|----------|----------|----------|--------|
| 1 | 提交订单 | POST /api/order | {参数} | {响应} | STOCK_INSUFFICIENT |

### {N}.2 异步流程

| # | 触发事件 | 处理链路 | 重试策略 | 补偿机制 |
|---|----------|----------|----------|----------|
| 1 | 支付回调 | 支付网关→订单服务→DB | 重试3次 | 人工对账 |

（可配合 sequenceDiagram 展示完整链路）
```

### 非功能需求章节（必选）

从业务逻辑清单的"边界条件清单"章节提取，每个版本必选。

```markdown
## {N}、非功能需求

### {N}.1 并发与幂等

| 场景 | 策略 | 实现方式 |
|------|------|----------|
| 重复提交订单 | 请求去重 | 按钮禁用 + 后端幂等键 |
| 并发抢购 | 乐观锁 | 库存预扣 + 最终校验 |

### {N}.2 数据一致性

| 场景 | 一致性要求 | 补偿机制 |
|------|------------|----------|
| 支付与订单状态 | 最终一致性 | 定时对账 + 告警 |

### {N}.3 性能与可用性

（按需补充：响应时间要求、超时设置、降级策略）
```

## 模块编排顺序

页面章节按**用户操作路径**排序，而非按文件名或页面层级排序：

### 推荐顺序

1. **账号与登录** — 登录、注册、微信登录（用户最先接触的流程）
2. **个人中心** — 个人资料、设置（登录后第一个管理页面）
3. **购物与商品** — 首页、搜索、商品详情、分类（核心购物路径）
4. **购物车** — 购物车管理
5. **收藏/浏览记录** — 辅助功能
6. **下单与支付** — 确认订单、支付、支付成功（转化路径）
7. **订单管理** — 订单列表、订单详情、退款售后（售后路径）

### 编排原则

- 同一操作流程的页面放在一起（登录+注册、确认订单+支付+支付成功）
- 先基础功能，后高级功能（先登录，再个人中心）
- 先主流程，后辅助功能（先购物车，再收藏）
- 同一模块的多个页面合并到同一个二级标题下

## 章节编号规则

```
# {项目名} - PRD V{版本号}              ← 一级标题，文档标题

## 一、版本概述                          ← 二级标题，章节
### 1.1 版本范围                         ← 三级标题，章节子项

## 二、账号与登录                        ← 二级标题，模块
### 2.1 手机号登录                       ← 三级标题，页面
#### 2.1.1 功能概述                      ← 四级标题，页面内小节
#### 2.1.2 页面结构
#### 2.1.3 操作流程
#### 2.1.4 字段与交互
#### 2.1.5 业务规则
#### 2.1.6 页面跳转

## 八、数据模型                          ← 二级标题，可选章节
## 九、页面导航                          ← 二级标题，必选章节
```

### 编号规则

- 章节：一、二、三...（中文数字）
- 模块内页面：2.1、2.2、2.3...
- 页面内小节：2.1.1、2.1.2、2.1.3...
- **所有标题必须带编号**，不允许出现无编号的标题
- 版本概述子节也必须编号：1.1 版本范围、1.2 核心功能、1.3 数据来源说明...
- 页面内非标准小节（如"两种模式""粘贴识别"）同样按顺序编号

## 与页面模板的关系

- 页面章节（`templates/prd-page-section.md`）定义**单个页面**的6节结构
- 本文件定义**整份版本PRD**的章节编排
- 两者配合使用：页面模板负责微观，结构规则负责宏观

## 常见错误

| 错误 | 正确做法 |
|------|----------|
| 按文件名字母排序页面 | 按用户操作路径排序 |
| 每个页面独立一个章节 | 同模块页面合并到一个二级标题下 |
| 省略版本概述 | 版本概述必选，帮助读者快速理解范围 |
| 数据模型放在文档开头 | 数据模型放在页面章节之后、页面导航之前 |
| 页面导航只列有跳转的页面 | 列出所有涉及页面，包括纯展示页 |
| 模块级定义写在某个页面内 | 独立为 {X}.0 前言小节 |
| 页面导航只有跳转表格 | 必须包含返回导航和登录拦截子节 |
| 编号使用阿拉伯数字(1. 2. 3.) | 章节用中文数字(一、二、三)，页面内用 X.Y.N |
| 忽略清单中的角色泳道图 | ≥2角色时生成"角色与架构"章节 |
| 忽略清单中的边界条件 | 边界条件清单转换为"非功能需求"章节（必选） |
| 忽略清单中的系统时序 | 跨系统时生成"接口时序"章节 |
