# 写作规则 V5.0

## 基本原则

### 1. 叙述体优先

- 功能概述必须用自然语言写，不用表格
- 操作流程先画流程图，再用1-2句话补充关键细节
- 页面结构先用1句话概括布局，再配表格

### 2. 信息不重复

同一条规则只能出现在一个地方：
- 字段校验规则 → 写在"字段与交互"表中
- 跨字段业务逻辑 → 写在"业务规则"表中
- 不要在交互说明、错误提示、业务规则中重复描述同一个校验

### 3. 表格使用原则

**适合用表格的场景：**
- 字段定义（名称、类型、规则、错误提示在一行）
- 业务规则（编号 + 描述）
- 页面跳转关系（来源 + 去向）

**不适合用表格的场景：**
- 页面整体描述 → 用叙述体
- 操作步骤说明 → 流程图 + 短文字
- 简单的入口出口关系 → 用列表

### 4. 去除冗余章节

以下章节在大部分小程序页面中不适用，不要生成：
- 数据权限设计
- 批量操作规则
- 操作日志要求
- 数据去向表（除非有明确的后端接口对接）

如果某个页面确实需要这些内容，按需添加。

## 可读性要求

1. 每个章节的开头用1-2句话概括本节内容，不要直接堆表格
2. 表格列数不超过6列
3. 流程图中的节点文案与正文用词保持一致
4. 错误提示文案使用引号包裹，如 Toast 提示"请输入手机号"
5. 规则编号格式：RULE-{模块缩写}-{序号}，如 RULE-LOGIN-001

## 规则编号规范

格式：`RULE-{MODULE}-{SEQ}`

| 组成 | 说明 | 示例 |
|------|------|------|
| MODULE | 模块英文缩写，2-6个字母大写 | AUTH、WX、REG、PROFILE、ADDR、FAV、SEARCH |
| SEQ | 三位数字，同一模块内递增 | 001、002、003 |

**同一模块的规则使用相同 MODULE 前缀**，不同模块绝不混用。跨模块的通用规则用 GUARD、COMMON 等通用前缀。

```
RULE-AUTH-001    ← 账号认证模块
RULE-WX-001      ← 微信登录模块
RULE-ADDR-001    ← 收货地址模块
RULE-GUARD-001   ← 登录守卫（跨模块通用）
```

## 与V2.0的关键差异

| 对比项 | V2.0 | V3.0 |
|--------|------|------|
| 功能概述 | 四行固定格式（功能/业务/角色/触发） | 2-4句自然语言 |
| 字段定义 | 拆成输出项表+输入项表+错误提示表 | 合并为一张表 |
| 不适用章节 | 保留标题写"不适用" | 直接省略 |
| 表格密度 | 高（几乎全表格） | 低（叙述+关键表格） |
| 重复度 | 高（同一规则出现4-5次） | 低（每条规则只出现1次） |
