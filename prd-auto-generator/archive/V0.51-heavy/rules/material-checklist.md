# 材料清单规则 V6.0

生成版本PRD前，按以下顺序读取原材料，确保信息完整。

V6.0 变更：更新业务逻辑清单的解析说明，适配 V0.22 的6列用例表、4列溯源三问、4个新增全局章节。

## 读取顺序

### 第一层：业务规格（必读）

按顺序读取，建立业务全貌：

| # | 材料 | 路径示例 | 作用 |
|---|------|----------|------|
| 1 | 业务逻辑清单 | `业务逻辑清单.md` 或 `业务逻辑清单_V{版本}.md` | 场景定义、操作步骤、预期结果、示例账号、统计口径、角色泳道图、单据流转、系统时序、边界条件 |
| 2 | 导航索引 | `page-index.md` 或 `page-index.html` | 页面层级关系、版本标记、页面列表 |
| 3 | 现有PRD文档 | `PRD/PRD-*.md` | 已有版本的PRD，避免重复，识别增量变更 |

### 第二层：源码实现（按页面读取）

仅读取本次版本范围内的页面源码：

| 类型 | 路径模式 | 作用 |
|------|----------|------|
| 页面HTML | `pages/{page_name}.html` | 页面结构、交互逻辑、字段定义、CSS样式 |
| 公共JS | `js/common.js` | 全局组件、通用工具函数 |
| 认证JS | `js/auth.js` | 账号体系、登录态管理、用户数据结构 |
| 认证守卫 | `js/auth-guard.js` | 受保护页面列表、拦截逻辑 |
| 公共CSS | `css/common.css` | 全局样式变量、组件样式 |

### 第三层：补充材料（按需读取）

| 材料 | 何时读取 | 作用 |
|------|----------|------|
| 会议纪要/讨论记录 | 用户提到"讨论过""开会定的" | 需求变更背景、决策理由 |
| 旧版PRD | 需要对比版本差异时 | 了解历史需求，识别变更点 |
| 外部API文档 | 页面涉及外部数据源时 | 接口定义、数据格式 |

## 版本范围确定

从 `page-index.md` 或 `page-index.html` 中提取带版本标记的页面，确定本次PRD覆盖范围。

示例：
```
- [我的](pages/profile.html) `V0.1`      ← 本次范围内
- [商品详情](pages/product_detail.html)    ← 不在范围内
- [搜索](pages/search.html) `V0.1`        ← 本次范围内
```

如果用户明确指定了页面列表，以用户指定为准。

## 读取策略

### 页面源码读取要点

读取每个页面HTML时，重点关注：

1. **页面标题和导航** — `<title>` 标签、`.nav-title` 文本
2. **表单字段** — `<input>`、`<textarea>`、`<select>` 的属性（type、placeholder、id）
3. **交互按钮** — 按钮文案、绑定的事件处理
4. **数据结构** — JS中的数据对象定义（数组、字段）
5. **校验逻辑** — 表单提交前的校验代码
6. **弹窗/面板** — modal、panel 的触发条件和内容
7. **页面跳转** — `location.href`、`history.back()` 的目标
8. **状态管理** — 登录态、本地存储的读写

### 公共模块读取要点

- `auth.js`：账号数据结构、默认值、CRUD方法
- `common.js`：通用组件（底部Tab、子页面导航）、工具函数
- `common.css`：`.phone-frame` 尺寸、全局颜色变量、组件样式类

## 信息整合规则

1. **业务逻辑清单优先** — 当源码实现与业务逻辑清单冲突时，以清单为准，在PRD中标注差异
2. **源码补充细节** — 业务逻辑清单未提及但源码中已实现的功能，纳入PRD
3. **版本标记对齐** — PRD中的页面范围必须与 index.md 中的版本标记一致
4. **数据来源标注** — 涉及商品数据时注明来源（品牌商城后台 / 魔方配置）

## 业务逻辑清单章节解析（适配 V0.22）

业务逻辑清单 V0.22 包含以下章节，解析时按目标PRD章节映射：

### 页面级章节

| 清单章节 | 清单格式 | PRD目标 | 解析要点 |
|----------|----------|---------|----------|
| 功能用例表 | 6列（场景/前置条件/操作步骤/预期结果/异常分支） | 页面章节：操作流程 + 字段与交互 + 业务规则 | 前置条件→页面结构的前置说明；操作步骤+预期结果→操作流程；异常分支→业务规则 |
| 关键字段数据来源 | 4列溯源三问（字段/从哪来/往哪去/变了怎么办） | 页面章节：字段与交互 | "从哪来"→来源列；"往哪去"→补充接口说明；"变了怎么办"→业务规则 |
| 业务逻辑增强 | 2列（问题/确认结果） | 页面章节：业务规则 | 确认结果→业务规则条目 |

### 全局章节

| 清单章节 | 清单格式 | PRD目标 | 解析要点 |
|----------|----------|---------|----------|
| 角色泳道图 | 角色×阶段矩阵表 | 角色与架构 | 提取角色列表→角色定义表；矩阵表→角色交互矩阵 |
| 核心单据流转 | 实体生命周期表 | 单据流转 | 直接映射为单据流转章节 |
| 系统交互时序 | 5列（触发操作/请求链路/同步异步/失败策略） | 接口时序 | 同步→同步接口表；异步→异步流程表 |
| 边界条件清单 | 4列（边界类型/场景/处理策略） | 非功能需求 | 按类型分组：并发/幂等→并发与幂等；一致性→数据一致性 |
| 页面导航 | 跳转关系+返回导航+登录拦截 | 页面导航 | 直接映射 |
| 状态流转 | 3列（当前状态/操作/目标状态） | 数据模型或独立章节 | 配合 stateDiagram 展示 |

## 常见错误

| 错误 | 正确做法 |
|------|----------|
| 只读源码不读业务逻辑清单 | 先读清单建立业务认知，再读源码补充细节 |
| 读取所有页面源码 | 只读版本范围内的页面 |
| 忽略公共JS中的数据结构 | auth.js 中的账号结构是关键输入 |
| 跳过现有PRD直接生成 | 先读现有PRD，避免重复或矛盾 |

## 增量版本处理

当生成新版本PRD（如V0.2）且已有旧版本PRD（如V0.1）时：

### 识别页面变更类型

| 类型 | 判断方式 | PRD处理 |
|------|----------|---------|
| 新增页面 | 旧版PRD中无对应章节 | 完整生成全部小节 |
| 变更页面 | 源码有修改或业务逻辑清单有调整 | 标注变更点，更新受影响小节 |
| 未变更页面 | 源码和业务逻辑均无变化 | 引用旧版，标注"同 V{上一版本}" |
| 删除页面 | 旧版有但新版范围不包含 | 不生成，在版本概述中说明移除 |

### 变更标注格式

变更页面在章节开头添加标注：

```
### 3.1 我的（profile.html）

> ⚠ V0.2 变更：新增浏览记录入口、调整菜单顺序

（后续内容只写变更部分，未变更部分标注"同 V0.1"）
```

### 规则编号衔接

- 旧版已有规则编号保留不变（如 RULE-AUTH-001 不变）
- 新增规则在对应模块下续编（如 RULE-AUTH-006）
- 变更规则保留原编号，更新描述
