# 写作规则 V6.0

**V6.0 变更**：编号体系对齐 UC/AC（见 structure-rules.md）；删去过时的版本差异对比；聚焦模块级 3 段写法。

---

## 基本原则

### 1. 叙述体优先

- 每段开头 1-2 句概括，再上表/图
- 主流程先画图，再用 1-2 句补充关键细节（不逐节点复述）
- 状态流转/异常/字段/CRUD 用表格

### 2. 信息不重复

同一条信息只出现在一个地方：

| 信息类型 | 写在哪里 |
|----------|----------|
| 字段校验规则 | 字段说明表「校验规则」列 |
| 字段错误提示 | 字段说明表「错误提示」列 |
| 跨字段业务逻辑 | 核心流程的异常与边界，或 RULE 编号 |
| 状态跳转条件 | 状态流转表 |

不要在叙述、表格、规则中重复描述同一条规则。

### 3. 表格使用原则

**适合表格**：字段定义、状态流转、异常边界、CRUD 矩阵、跳转关系

**不适合表格**：模块整体描述、操作流程说明 → 用叙述体或流程图

### 4. 按需生成

不适用子节直接省略，不留空占位、不写"不适用"。以下在大部分场景不生成，确有需要再补：

- 数据权限设计、批量操作规则、操作日志要求

---

## 可读性要求

1. 每段开头 1-2 句概括，不直接堆表格
2. 表格列数 ≤ 9（字段说明表上限 9 列，首列字段中文名）
3. 流程图节点文案与正文用词一致
4. 错误提示文案用引号包裹：`提示「请输入手机号」`
5. 预期结果/Then 具体到状态、文案、跳转，不写"显示正确""操作成功"

---

## 编号规范

主编号体系见 `structure-rules.md`：

| 对象 | 格式 |
|------|------|
| 用例 | `UC-{代号}-{序号}` |
| 验收 | `AC-{代号}-{序号}` |
| 业务规则 | `RULE-{代号}-{序号}` |

> UC/AC 是主链路（用例↔验收强绑定）。RULE 仅用于跨字段、非显而易见的业务规则补充编号，不替代 UC/AC。

---

## 叙述体 vs 表格（分工）

| 内容 | 写法 |
|------|------|
| 操作步骤 / 交互流程 | 叙述体 + 流程图 |
| 字段 / 枚举 / 校验规则 | 表格 |
| 页面跳转关系 | 列表或跳转表 |
| 状态流转 | 状态表 + 流程图 |

**核心**：叙述体说 why 和整体，表格说 what 和细节，两者不重复。
